## Architecture native

### Protocole JS ↔ natif

Le module expose 5 méthodes via TurboModule :

| Méthode | Type | Rôle |
|---|---|---|
| `getEnergyState()` | `Promise<EnergyStatePayload>` | Lecture one-shot de l'état courant |
| `startMonitoring()` | `void` | Active les observers natifs |
| `stopMonitoring()` | `void` | Désactive les observers natifs |
| `addListener(name)` | `void` | Requis par `NativeEventEmitter` |
| `removeListeners(n)` | `void` | Requis par `NativeEventEmitter` |

L'événement émis s'appelle `"EnergyAware:stateChanged"` et transporte le même payload que `getEnergyState()` :

```ts
interface EnergyStatePayload {
  batteryLevel: number;    // 0.0–1.0, -1 si inconnu
  batteryState: string;    // 'charging' | 'unplugged' | 'full' | 'unknown'
  isLowPowerMode: boolean;
  thermalState: string;    // 'nominal' | 'fair' | 'serious' | 'critical'
}
```

### Ref-counting du monitoring

`startMonitoring()` / `stopMonitoring()` sont ref-countés côté natif (`_monitoringCount` sur iOS, `monitoringCount` sur Android). Le monitoring physique (battery monitoring, notifications, receivers) ne s'active qu'au premier appel et ne se désactive qu'au dernier. Cela permet à plusieurs instances de `useEnergyState` de coexister sans conflits.

### iOS (`EnergyAware.mm`)

- Classe : `EnergyAware : RCTEventEmitter <NativeEnergyAwareSpec>`
- Base `RCTEventEmitter` pour compatibilité old arch + new arch via interop layer
- `getTurboModule:` retourne `NativeEnergyAwareSpecJSI` pour new arch
- Notifications observées :
  - `UIDeviceBatteryLevelDidChangeNotification`
  - `UIDeviceBatteryStateDidChangeNotification`
  - `NSProcessInfoPowerStateDidChangeNotification`
  - `NSProcessInfoThermalStateDidChangeNotification`
- `batteryMonitoringEnabled = YES` activé dans `startMonitoring`, désactivé dans `stopMonitoring`
- `_hasListeners` guard : les events ne sont émis que si un listener JS est actif

### Android (`EnergyAwareModule.kt`)

- Classe : `EnergyAwareModule : NativeEnergyAwareSpec`
- Thermal state via `PowerManager.currentThermalStatus` (API 29+), fallback `"nominal"` sur API < 29
- 3 `BroadcastReceiver` séparés : `ACTION_BATTERY_CHANGED`, `ACTION_POWER_SAVE_MODE_CHANGED`, `ACTION_THERMAL_STATUS_CHANGED`
- Batterie lue via sticky broadcast (`registerReceiver(null, filter)`) sans receiver persistant pour les lectures one-shot
- Emission via `DeviceEventManagerModule.RCTDeviceEventEmitter`
- `addListener` / `removeListeners` sont des no-ops (Android n'en a pas besoin)

## Logique `energyMode`

Fonction pure dans `src/utils/computeEnergyMode.ts`.

Priorité décroissante :

1. **`overheating`** — thermalState `serious` ou `critical` (toujours, même en charge)
2. **`critical`** — batterie < 15%, OU (low power mode ET batterie < 20%) — seulement si non en charge
3. **`saver`** — low power mode actif, OU thermalState `fair`, OU batterie < 30% non en charge
4. **`normal`** — sinon

Cas notable : `batteryLevel = -1` (iOS quand monitoring non encore actif) est ignoré dans les seuils (`batteryLevel >= 0` requis).

## Hook `useEnergyState`

```
mount
  → startMonitoring()          // active les observers natifs
  → getEnergyState()           // lit l'état initial (async)
  → NativeEventEmitter.addListener('EnergyAware:stateChanged', …)

event reçu
  → toEnergyState(raw)         // cast types + calcule energyMode
  → setState(…)

unmount
  → subscription.remove()
  → stopMonitoring()           // décrémente le ref-count natif
```

`mountedRef` protège le `setState` de la promise initiale contre les unmounts rapides.

## Ce qui N'est PAS dans cette version

- Pas de `EnergyProvider` / Context React — chaque hook monte ses propres listeners. Pour des apps avec beaucoup de composants utilisant `useEnergyState`, un provider sera nécessaire en v0.2.
- Pas de composants drop-in (`EnergyAwareImage`, `EnergyAwareAnimation`, etc.) — prévu v0.2
- Pas de `useAdaptiveInterval` — prévu v0.2
- Pas de debug overlay — prévu v0.2
- Pas de support TurboModule bridgeless pur (bridge interop requis pour iOS EventEmitter) — acceptable pour RN 0.85

## Points de vigilance pour les prochaines versions

- **EnergyProvider** : quand plusieurs composants utilisent `useEnergyState`, chacun appelle `startMonitoring`. Le ref-count natif évite les doublons d'observers, mais il y a autant d'abonnements `NativeEventEmitter` que de composants. Un context React permettrait de n'en avoir qu'un.
- **Codegen** : modifier `NativeEnergyAware.ts` nécessite de relancer le codegen (`pod install` + gradle build) pour régénérer `EnergyAwareSpec.h` / `NativeEnergyAwareSpec.kt`.
- **batteryLevel sur iOS** : vaut `-1` si `batteryMonitoringEnabled` est `false`. Notre `startMonitoring` l'active avant `getEnergyState`, donc en pratique ce cas ne se présente pas — mais `computeEnergyMode` le gère via le guard `>= 0`.
- **Thermal sur Android < API 29** : retourne toujours `"nominal"`. Pas de fallback heuristique pour les vieux devices — décision volontaire de garder la lib simple.
